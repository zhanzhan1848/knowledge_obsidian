## 研究问题
OpenFOAM 配置门槛高：需要组装多目录输入文件（求解器、离散化、边界条件字典），非专业用户难以使用。现有 LLM+RAG 系统存在三个局限：
1. 单次平面查询检索
2. 单一检索策略处理所有请求
3. 单 Agent 既当编写者又当审查者

## 核心贡献：IteraSim RAG 架构

### 三阶段检索流程
```
用户查询 → LLM 查询扩展 → 多策略融合 → HNSW 向量检索
         ├─ 物理变体 (physics variants)
         ├─ 求解器关键词变体 (solver-keyword)
         └─ 故障排除变体 (troubleshooting)
         
         → Reciprocal Rank Fusion (RRF) 合并排名
         → Maximal Marginal Relevance (MMR) 重排序
```

### 检索路径分离
- **工具条件工作流查询**: 定向检索路径
- **语料库物理查询**: 独立检索路径
- **确定性关键词路由器** 调度

### 三 Agent 协作
| Agent | 职责 |
|-------|------|
| Architect | 规划求解器/离散化方案 |
| InputWriter | 生成 OpenFOAM 输入文件 |
| Reviewer | 审查并修复错误（bounded loop, max 10 次）|

### 静态知识层
覆盖：
- 求解器选择原则
- 湍流闭合模型
- 边界条件规范
- 有限体积默认值

## 基准测试结果 (28 cases)
| 类别 | 检索覆盖率 |
|------|----------|
| Zero-shot setup | ~77.9% (mean) |
| Few-shot generalization | ~79.1% (median) |
| Single-parameter modification | **>90%** |
| Turbulence-model swaps | ~77.9% |

**关键指标**:
- 6 个参考配置全部在 OpenFOAM v2506 上运行成功
- 2 个故意损坏案例在 Reviewer loop 中被诊断并修复
- Benchmark + 评分规则 + 损坏案例探针已开源

## 与现有系统对比
| 系统 | 检索 | Agent分离 | Loop bound |
|------|-----|----------|-----------|
| OpenFOAMGPT | 平面 | 否 | 开放 |
| MetaOpenFOAM | 平面 | 是 (3 roles) | - |
| Foam-Agent 1.0 | 平面 | 是 | 25 |
| Foam-Agent 2.0 | 分层多索引 | 是 | 25 |
| ChatCFD | 平面+错误定位 | 是 | - |
| **IteraSim RAG** | **多阶段** | **是 (3)** | **10** |

## 关键创新
1. **多阶段检索**: LLM 查询扩展 → RRF 融合 → MMR 重排
2. **三 Agent 架构**: 职责分离，审查有界
3. **工具感知路由**: 不同类型查询走不同检索路径
4. **破坏性测试**: 合成损坏案例验证修复能力

## 软件工程价值
- 降低 OpenFOAM 使用门槛
- 自动化错误诊断（仅需求解器日志）
- 加速 CFD 案例配置

## 可行性分析
- ✅ **开源**: 面向 OpenFOAM (开源 CFD)
- ✅ **实用性**: 直接对接工业 CFD 工作流
- ✅ **可复现**: Benchmark + 代码已发布
- ⚠️ **依赖 LLM API**: 商业部署需考虑成本
