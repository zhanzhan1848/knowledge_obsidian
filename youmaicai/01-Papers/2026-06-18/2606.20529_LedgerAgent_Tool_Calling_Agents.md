## 核心贡献
1. 提出 **LedgerAgent**：一种推理时方法，通过独立的 ledger 维护观察到的任务状态，并将其渲染到 prompt 中
2. 解决 Agent 的两个常见失败模式：
   - Agent 可能检索到正确事实，但后来基于过时/缺失/错误信息做出决策
   - 语法有效的工具调用仍可能违反依赖当前任务状态的领域策略
3. Ledger 还用于在执行环境变更工具调用前检查状态依赖的策略约束，阻止策略违规
4. 在四个客户服务领域和多种开源/闭源模型上验证，平均 pass@k 提升

## 方法

### 问题
标准 Agent 中任务状态不单独表示：观察、工具返回值、策略指令都放在 prompt 中，导致 Agent 每次决定下一步时都需从 prompt 重建相关状态 → 状态管理是隐式的

### LedgerAgent 架构
1. **独立 Ledger**：维护跨轮次观察到的任务事实、标识符、约束、条件
2. **状态渲染**：将 ledger 状态渲染到 prompt，而非依赖 Agent 隐式重建
3. **策略检查**：执行环境变更工具调用前，用 ledger 检查状态依赖的策略约束

### 任务状态构成
- 事实（Facts）
- 标识符（Identifiers）
- 约束（Constraints）
- 条件（Conditions）

## 实验结果
- 四个客户服务领域测试
- 混合开源/闭源模型面板
- 平均 pass@k 提升，最大收益来自更严格的多轮一致性指标
- 相比标准 prompt-based 工具调用方法有显著改进

## 局限性
- 目前仅在客户服务场景验证
- 状态提取和 ledger 维护本身可能出错

## 建议
- **是否推荐使用**：是（对生产环境 LLM Agent 开发有直接价值）
- **适用场景**：客服 Agent、工具调用系统、Policy-compliant AI
- **关键词**：LLM agent, tool calling, policy adherence, state management
