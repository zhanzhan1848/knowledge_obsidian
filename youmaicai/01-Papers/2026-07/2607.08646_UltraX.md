## 核心贡献

1. **问题**：随着缩放定律收益递减，LLM 性能提升从"数据扩展"转向"数据质量提升"，但大规模语料下的现有方法存在质量-效率-可靠性三角困境
2. **方案**：提出 **UltraX**，一种函数调用式大规模预训练数据精炼框架，引入插入操作补全编辑函数空间（Insertion + Deletion + Modification）
3. **创新**：构建可靠的程序监督生成管道，包括数据集自适应的 Prompt 优化、Line Alignment Mapping、Dynamic Context Replacement

## 核心方法

### 函数调用式编辑优势
- **对比端到端生成**：E2E 生成需输出与输入规模成正比的文本，推理开销高；函数调用仅输出结构化操作序列，Token 数大幅减少
- **自然兼容分段处理**：适合处理长文档的分段-重组，保证语义连贯性

### 三大模块

#### 1. 可靠的种子监督生成管道
```
专家LLM生成高质量端到端精炼文本
         ↓
Line Alignment Mapping (原始文本-精炼文本对 → 结构化操作序列)
         ↓
Dynamic Context Replacement (动态上下文替换)
         ↓
低置信度过滤 + 操作组合比例控制采样
```

#### 2. 完整的编辑函数空间
| 操作 | 说明 |
|------|------|
| Deletion | 删除低质量片段 |
| Modification | 修改错误/噪声内容 |
| **Insertion** | 插入缺失/补充信息（UltraX 新增）|

#### 3. 鲁棒的大规模执行管道
- Sliding-window prediction（滑动窗口预测）
- Global operation aggregation（全局操作聚合）
- 系统性后处理：歧义替换过滤、邻接操作合并、重复模式回退

### 与 ProX/RefineX 对比
| 方法 | 函数空间 | 种子监督 | 执行可靠性 |
|------|------|------|------|
| ProX | 仅删除 | LLM直接生成 | 存在问题 |
| RefineX | 不完整 | 受限 | 存在问题 |
| UltraX | 完整（增/删/改） | 层级映射 + 专家LLM | 系统性后处理 |

## 实验结果

### 设置
- 1B 参数 MiniCPM，从头预训练，20B Token 训练预算
- 语料: FineWeb, RedPajama-V2, AICC, Ultra-FineWeb, FineWeb-ProX-Doc

### 结果
- **UltraX 在所有语料上平均性能最高**，多个数据集相对提升超过 2%
- **更强的数据效率**：用更少训练 Token 达到同等或更好性能
- 收益来自细粒度精炼（噪声去除与信息保留的更好平衡）

## 局限性
- 依赖专家 LLM 生成种子监督，计算成本较高
- 插入操作的编辑质量受限于语言模型的语义理解能力
