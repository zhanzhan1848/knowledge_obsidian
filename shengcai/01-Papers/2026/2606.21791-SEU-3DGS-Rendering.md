## 核心贡献
1. 首次系统研究 3DGS 渲染中的单粒子翻转 (SEU) 效应
2. GPU 并行故障注入引擎：380 万次受控单比特翻转，覆盖 4 场景、6 字段、所有比特位置、3 种数值格式 (fp32/fp16/bf16)，耗时 5.3 GPU 小时
3. 发现故障高度集中：少数高阶位（尤其对数 scale 的符号位）可将单个图元扩大到覆盖 75.7% 帧面积
4. 从 IEEE-754 布局和流水线激活推导封闭形式扰动边界，预测每位故障影响
5. 提出 Support Guard：逐图元将参数 clamp 到训练时观察到的坐标框，额外开销仅 76μs/帧
6. 在 20000 个并发翻转下，受保护渲染器保持 21.8dB vs 无保护 10.6dB

## 技术方案

### 问题定义
3DGS 作为标准实时场景表示，越来越多部署在暴露于瞬态故障的硬件上（空间处理器、机器人边缘设备），其中单粒子翻转导致静默数据损坏。

### 核心方法
1. **GPU 并行故障注入引擎**: 高效模拟单比特翻转

2. **位级敏感度分析**:
   - 大多数翻转因高冗余感知不变
   - 少数高阶位导致灾难性帧覆盖错误

3. **封闭形式扰动边界**:
   - 基于 IEEE-754 布局和流水线激活
   - 预测每位翻转的影响排序

4. **Support Guard 防御**:
   ```
   For each primitive p:
     For each parameter f in p:
       f_clamped = clamp(f, training_bbox[f])
   ```

### 关键数据
- 单比特翻转影响高度集中
- 75.7% 帧覆盖（最坏情况）
- 760μs/帧开销（Guard）
- 20000 并发翻转: 无保护 10.6dB vs 有保护 21.8dB

## 实验结论
- 3DGS 对硬件故障高度敏感
- Support Guard 有效防御，额外开销可接受
- 封闭形式边界可预测故障影响
- 分布式渲染中 Guard 可限制污染范围

## 局限性
- 主要关注 GPU 单粒子翻转，未涉及 CPU 或内存错误
- Guard 可能影响某些正常情况下的合法变形

## 可行性分析
- **实现难度**: 低
  - Support Guard 实现简单（逐 primitive clamp）
  - 已有开源数据集
- **性能预期**: 优秀
  - 仅 76μs/帧开销
  - 有效防止灾难性故障
- **适用场景**: 空间渲染、边缘计算、自动驾驶安全关键系统
